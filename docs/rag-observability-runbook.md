# RAG 运行观测 Runbook

本文用于线上或预发环境排查 RAG 问答质量、响应慢、流式输出中断和工具调用异常。建议和 `RAG Trace 阅读指南`、`检索排查指南` 一起使用。

## 观测入口

| 入口 | 关注内容 |
|---|---|
| 应用日志 | 请求入口、模型调用异常、检索异常、MQ 投递和消费失败。 |
| RAG Trace | 单次问答的意图识别、检索、Prompt、模型调用和工具调用节点。 |
| 管理后台 | Trace 查询、知识库文档状态、系统设置和模型配置。 |
| 数据库 | 会话、消息、文档、分块、Trace run/node 和调度执行记录。 |
| 中间件控制台 | Redis 锁与缓存、RocketMQ 消费堆积、PostgreSQL 慢查询、Milvus collection 状态。 |

## 问答链路基线

一次正常知识库问答通常包含：

1. 前端发起 SSE 请求。
2. 后端创建或读取会话上下文。
3. 会话记忆拼接历史摘要。
4. 问题改写和术语映射。
5. 意图识别或澄清判断。
6. 多通道检索并合并结果。
7. 去重、Rerank 和上下文格式化。
8. Prompt 构建并调用模型。
9. 流式 token 写回前端。
10. Trace、消息和反馈数据落库。

排查时先确认链路停在哪一步，不要直接从模型输出质量开始猜测。

## 常见症状

### SSE 没有输出

- 查前端请求是否命中 `/api/ragent` 后端路径。
- 查后端是否创建 SSE emitter，是否被限流或排队拒绝。
- 查模型首包探测是否失败，失败后是否尝试了候选模型。
- 查浏览器代理或网关是否缓冲了流式响应。

### 回答没有引用知识库

- 查 Trace 中是否进入知识库意图或全局检索通道。
- 查检索结果数量和分数是否为空或过低。
- 查文档是否已成功分块并写入当前启用的向量库。
- 查 Prompt 上下文是否被 token 限制截断。

### 回答慢

- 分别记录检索耗时、Rerank 耗时、模型首包耗时和总生成耗时。
- 检查向量库慢查询、模型供应商延迟和 RocketMQ 消费堆积。
- 如果只有首包慢，重点看模型路由和供应商健康状态。
- 如果只有知识库问题慢，重点看检索通道数量和 topK 设置。

### 工具调用异常

- 查意图是否被识别为工具类意图。
- 查 MCP 工具是否已注册到 `McpToolRegistry`。
- 查参数提取 Prompt 的输入和输出 JSON。
- 查远程 MCP server 地址、接口路径和响应格式。

## 最小排查字段

记录问题时至少保留：

- 请求时间、用户、会话 ID 和消息 ID。
- Trace run ID。
- 使用的模型 provider、model name 和候选顺序。
- 检索通道、命中文档 ID、chunk ID 和分数。
- Prompt 场景和最终上下文长度。
- SSE 是否首包成功、是否被用户停止。

这些字段能把一次问题从“回答不对”还原成可定位的工程事件。

## 发布后巡检

每次发布涉及 RAG 编排、检索、模型路由或 MCP 时，建议抽查：

1. 普通知识库问答可以完整生成。
2. 低置信度问题会进入澄清或全局检索兜底。
3. 模型候选失败后会切换到下一个候选。
4. Trace run 和节点能在管理后台查询。
5. 前端停止生成后后端任务能尽快释放。
6. 工具类问题不会误触发知识库检索。
