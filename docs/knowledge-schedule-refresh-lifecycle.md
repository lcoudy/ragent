# 知识库定时刷新生命周期

本文说明知识库文档定时刷新的执行链路，适合在排查“文档为什么没有按计划更新”“任务为什么卡住”“重复刷新如何避免”时参考。

## 核心对象

| 对象 | 作用 |
|---|---|
| `KnowledgeDocumentScheduleDO` | 保存文档刷新计划，包括 cron、状态、下一次运行时间和执行配置。 |
| `KnowledgeDocumentScheduleExecDO` | 记录每一次调度执行的开始、结束、结果和失败原因。 |
| `ScheduleRefreshProcessor` | 执行单个刷新任务的核心处理器，负责状态推进和异常收敛。 |
| `ScheduleLockManager` | 基于 Redis 控制同一计划的并发执行，避免多实例重复刷新。 |
| `KnowledgeDocumentChunkEvent` | 文档重新分块后投递到 RocketMQ 的异步事件。 |

## 执行入口

`KnowledgeDocumentScheduleJob` 定期扫描到期计划，并把单个计划交给 `ScheduleRefreshProcessor`。处理器会先读取计划和文档状态，再进入受控的状态流转。生产环境中需要保证：

- 应用实例时间同步，避免 cron 判断出现明显偏差。
- Redis 可用，否则分布式锁无法正常获取。
- RocketMQ 可用，否则文档刷新可能完成解析但无法进入后续分块消费。

## 状态流转

典型刷新流程如下：

1. 扫描到 `nextRunTime` 已到期的启用计划。
2. 使用 `ScheduleLockManager` 获取计划级锁。
3. 创建或更新执行记录，标记本次刷新开始。
4. 根据文档来源重新获取内容，例如本地文件、HTTP 地址或对象存储。
5. 创建入库任务或触发文档重新解析。
6. 投递分块事件到 RocketMQ。
7. 更新执行记录和下一次运行时间。
8. 释放锁。

如果任一步失败，处理器会记录失败原因，并尽量把计划推进到可重试状态。排查时优先看执行记录，再看应用日志和 MQ 消费日志。

## Cron 与频率保护

`CronScheduleHelper` 负责计算下一次执行时间，并判断计划间隔是否过短。新增或修改计划时建议：

- 使用标准 cron 表达式，避免依赖本地时区隐含假设。
- 不要把刷新间隔设置得过短，文档解析、embedding 和向量写入都可能消耗外部资源。
- 修改 cron 后立即检查下一次运行时间是否符合预期。

## 并发控制

定时刷新支持多实例部署，但同一个计划同一时间只能被一个实例处理。锁的关键点：

- 锁粒度应绑定计划或文档，避免不同文档互相阻塞。
- 锁需要设置租约时间，防止实例异常退出后永久占用。
- 处理耗时较长时需要关注锁过期后的重复执行风险。

如果发现同一文档被重复刷新，先检查 Redis 锁 key、租约时间和应用实例日志。

## 与入库 Pipeline 的关系

定时刷新不是单独的解析实现，而是把“按计划触发”接入已有入库链路。真正的解析、清洗、分块、增强、向量化仍复用入库 Pipeline 和知识库处理逻辑。因此：

- Pipeline 节点失败会反映为刷新执行失败或文档状态异常。
- 分块策略、embedding 模型和向量库配置变更会影响刷新结果。
- 定时刷新适合重新拉取外部内容，不适合绕过入库校验直接写向量库。

## 排查顺序

1. 查计划是否启用，cron 是否能计算出下一次运行时间。
2. 查 `nextRunTime` 是否已经到期，系统时间是否正确。
3. 查 Redis 锁是否被其他实例持有或遗留。
4. 查执行记录中的失败原因。
5. 查文档来源是否可访问，凭据是否过期。
6. 查 RocketMQ 是否成功投递和消费分块事件。
7. 查文档状态、分块数量和向量库写入结果。

## 发布检查

涉及定时刷新代码变更时，建议至少覆盖：

- cron 空值、非法值和正常值计算。
- 获取锁失败时不会重复执行。
- 文档来源失败时执行记录可见。
- 成功刷新后下一次运行时间会更新。
- RocketMQ 投递失败时不会误标记为完全成功。
