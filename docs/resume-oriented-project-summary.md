# Ragent 项目简历复盘摘要

本文面向准备把 Ragent 写进简历或面试复盘的贡献者，帮助把“做了一个 RAG 项目”拆成可解释、可追问、可验证的工程经历。

## 一句话项目描述

Ragent 是一个基于 Java 的全栈 Agentic RAG 平台，覆盖文档入库、解析分块、向量化、向量检索、重排、意图识别、多轮会话记忆、模型路由、MCP 工具调用、链路追踪和管理后台。

可以在简历中写成：

> 参与 Ragent AI 智能问答平台建设，负责 RAG 检索链路、模型路由、文档入库 Pipeline、链路追踪和工程化文档/测试补强，提升系统可排查性和持续演进能力。

## 架构分层

- `bootstrap`：主业务应用，包含聊天、RAG 编排、知识库、入库 Pipeline、管理后台和用户体系。
- `framework`：通用基础设施，包括统一响应、异常、Redis/cache、SSE、幂等、用户上下文和 RAG Trace 上下文。
- `infra-ai`：模型供应商抽象层，负责 Chat、Embedding、Rerank、模型选择、健康状态和故障切换。
- `mcp-server`：独立 MCP server，提供天气、票务、销售等工具示例。
- `frontend`：Vite React 管理后台和聊天界面，按 service、store、page、component 分层。

面试中可以强调：业务层不直接写供应商 HTTP 细节，而是通过 `infra-ai` 的抽象调用模型能力，后续新增供应商不会污染 RAG 编排代码。

## 核心链路

### 文档入库

```text
上传/调度文档
  -> IngestionEngine
  -> Fetcher / Parser / Chunker / Enhancer / Enricher / Indexer
  -> KnowledgeChunk
  -> Embedding
  -> pgvector 或 Milvus
```

可讲重点：

- Pipeline 通过 `PipelineDefinition` 和 `NodeConfig` 表达节点顺序与条件。
- `IngestionNode` 作为扩展点，新增节点不需要修改引擎主体。
- 节点日志记录每个节点的执行结果、耗时、异常和输出摘要，方便定位入库失败。

### 智能问答

```text
用户问题
  -> 会话记忆加载
  -> 问题改写和子问题拆分
  -> 意图识别
  -> 多通道检索
  -> 去重和 Rerank
  -> Prompt 组装
  -> 模型路由调用
  -> SSE 流式返回
```

可讲重点：

- 多通道检索同时支持意图定向检索和全局向量兜底，兼顾精准和召回。
- 后置处理器先去重再 Rerank，减少重复候选对排序和上下文窗口的污染。
- 会话记忆支持摘要和最近历史结合，避免无限增长导致 prompt 失控。

### 模型路由

模型调用通过候选列表、默认模型、优先级和健康状态共同决定实际调用目标。失败时 `ModelRoutingExecutor` 会标记失败并尝试下一个候选；达到阈值后进入熔断窗口，后续请求会跳过不可用模型。

可讲重点：

- Chat、Embedding、Rerank 共用同一套路由和健康状态逻辑。
- 流式 Chat 在首包阶段做失败探测，避免长时间占用 SSE 连接。
- provider 差异被收敛在 `infra-ai`，业务服务只关心能力接口。

### RAG Trace

RAG Trace 通过注解和上下文记录每次问答的根链路和节点执行信息，落库后可通过管理后台或接口查询。

可讲重点：

- 根节点记录问题、用户、会话、任务、状态和耗时。
- 子节点记录改写、意图、检索、生成等阶段。
- 结合检索排查指南，可以快速判断问题发生在召回、排序还是生成阶段。

## 个人贡献可讲点

- 补充本地启动、数据库初始化、前端联调、MCP server 和模型供应商配置文档，降低新人接入成本。
- 梳理多通道检索、RAG Trace、模型路由、知识库操作和检索排查路径，提高系统可维护性。
- 为入库 Pipeline 校验、检索通道启用、检索结果去重、模型路由故障切换和术语映射工具补充单元测试。
- 维护队列式贡献流程，用独立 commit 拆分每天可发布的真实改进，避免空提交和无意义贡献。

## 面试追问准备

- 为什么要做多通道检索，而不是只做向量检索？
- 意图识别失败时如何兜底？
- Rerank 在什么位置执行，为什么不在去重前执行？
- 模型供应商异常时如何故障切换？
- 文档入库失败如何定位到具体节点？
- 会话历史如何控制长度，摘要什么时候触发？
- pgvector 和 Milvus 的取舍是什么？
- MCP 工具调用和知识库检索如何协同？

## 后续改造方向

- 引入关键词检索或混合检索，补足纯向量召回对精确术语和编号的弱点。
- 对检索链路增加命中率、召回耗时和 Rerank 排序变化统计。
- 增强入库 Pipeline 的可视化编辑和节点级重试能力。
- 将模型健康状态暴露到管理后台，便于观察供应商故障和熔断恢复。
- 增加端到端 RAG 评测集，用固定问题集跟踪召回质量和答案稳定性。
